Create `.claude/skills/agency/questioning-flow.md`. This skill is loaded by `/agency:start` and provides the complete questioning methodology.

Format:
```yaml
---
name: agency:questioning-flow
description: Adaptive project initialization questioning that captures vision, requirements, and preferences
---
```

Structure the skill with these sections:

**Section 1: Questioning Philosophy**
Brief statement of principles:
- Vision first, technology second
- Adaptive depth — follow the user's energy
- Infer where possible, confirm where uncertain
- Structured choices for preferences, open conversation for vision
- Aim for 5-8 exchanges total (not 20 questions)

**Section 2: Conversation Flow**

The flow has 3 stages. Each stage lists the CORE questions (must ask) and OPTIONAL questions (ask if relevant). The flow should feel like a conversation, not a form.

**Stage 1: Vision & Identity** (1-3 exchanges)
Core:
- "What are you building? Give me the elevator pitch."
- "Who is this for? Who are the primary users?"
Optional (infer or ask based on response):
- "What problem does this solve?" (if not clear from elevator pitch)
- "What exists today that's close to this?" (competitive context)
- "What's the one thing that makes this different?" (value prop)

After Stage 1, summarize what you heard and confirm: "Here's what I'm understanding: [summary]. Anything to correct or add?"

**Stage 2: Requirements & Constraints** (2-4 exchanges)
Core:
- "What are the must-have features for v1?" (active requirements)
- "What's explicitly out of scope?" (boundaries)
Optional:
- "Any technical constraints or preferences?" (stack, platform, hosting)
- "What are the biggest risks or unknowns?" (constraints)
- "Any existing code, design, or content to work with?" (brownfield check)
- "Are there deadlines or milestones that matter?" (time constraints)

Adapt based on project type:
- **Code project**: Ask about stack, frameworks, deployment
- **Content project**: Ask about platforms, voice/tone, frequency
- **Design project**: Ask about brand, target devices, accessibility needs
- **Mixed project**: Cover the most relevant areas, skip others

**Stage 3: Workflow Preferences** (1-2 exchanges)
Use AskUserQuestion tool with structured choices:

Question 1 — Execution Mode:
- **Guided** (Recommended): Agency recommends, you approve before each step
- **Autonomous**: Agency plans and executes, you review at checkpoints
- **Collaborative**: Work alongside agents, high interaction

Question 2 — Planning Depth:
- **Standard** (Recommended): Balanced planning with 3-6 phases
- **Quick Sketch**: Minimal planning, get building fast (2-3 phases)
- **Deep Analysis**: Thorough research and planning before any execution (6+ phases)

Question 3 — Cost Profile:
- **Balanced** (Recommended): Opus for planning, Sonnet for execution, Haiku for checks
- **Economy**: Sonnet for everything, Haiku for checks
- **Premium**: Opus for planning and execution, Sonnet for checks

**Section 3: Output Structure**

After questioning is complete, the skill produces a structured data block that maps directly to the template placeholders. Document the exact mapping:

```
## Output Mapping

### PROJECT.md Placeholders
- {project_name} ← from Stage 1 (project title)
- {project_description} ← from Stage 1 (2-3 sentence description)
- {value_proposition} ← from Stage 1 (core value / differentiator)
- {target_users} ← from Stage 1 (who it's for, bulleted)
- {requirements_list} ← from Stage 2 (bulleted, prefixed with "- [ ]")
- {out_of_scope} ← from Stage 2 (bulleted)
- {constraints} ← from Stage 2 (bulleted)
- {decisions_table} ← from Stages 1-3 (key decisions made during questioning)
- {architecture_notes} ← from Stage 2 (technical choices, if any)
- {date} ← current date

### ROADMAP.md Placeholders
- {project_name} ← same as PROJECT.md
- {phase_checklist} ← generated from requirements decomposition
- {phase_details_sections} ← generated per phase with goals, requirements, agents, criteria
- {progress_table} ← initialized with all phases at 0 completed

### STATE.md Placeholders
- {total_phases} ← count of phases in roadmap
- {total_plans} ← estimated total plans (2-3 per phase)
- {progress_bar} ← initialized empty
- {progress_percent} ← 0
- {recent_decisions} ← from questioning (mode, depth, cost profile)
- {first_phase_name} ← name of phase 1
- {date} ← current date
```

**Section 4: Phase Decomposition Guidelines**

How to break requirements into phases for ROADMAP.md:
1. Group related requirements by dependency and domain
2. Order phases by dependency — foundational work first, user-facing features later
3. Each phase should be completable in 2-3 plans (max 3 tasks per plan)
4. Name phases descriptively: "API Foundation", "User Authentication", "Dashboard UI"
5. Assign recommended agents per phase using the agent registry's recommendation algorithm
6. Include success criteria that are testable/observable — not vague ("it works")
7. Estimate plan count: simple phases = 1-2, standard = 2-3, complex = 3

**Section 5: Edge Cases**

- **User wants to skip questions**: Infer reasonable defaults, note assumptions in decisions table
- **User has a PRD/spec**: Parse it instead of asking — extract project name, requirements, constraints. Skip to Stage 3 (workflow preferences)
- **Existing .planning/ directory**: Warn user that re-running /agency:start will overwrite. Confirm before proceeding.
- **Unclear project scope**: Push for specificity — "That's broad. What's the ONE thing you'd ship first?"
- **Non-code project**: The Agency handles marketing, design, content — adapt questioning to skip technical stack questions
