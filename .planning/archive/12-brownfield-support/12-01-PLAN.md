**Section 1: Principles & Detection** — When and how to determine if brownfield analysis applies:

```
### When to Run
- Triggered by /agency:start when existing source code is detected in the project directory
- Can be re-triggered manually if the codebase has changed significantly
- NEVER runs automatically — always prompted via AskUserQuestion

### Source Code Detection Heuristic
Check for non-Agency files in the current directory:
1. Any source files outside .planning/ and .claude/?
   Glob("*.{ts,js,py,rb,go,rs,java,swift,kt,c,cpp,cs}")
   Glob("src/**", "app/**", "lib/**", "components/**")
2. Any dependency manifests at root?
   package.json, Gemfile, pyproject.toml, requirements.txt, go.mod, Cargo.toml, pom.xml
3. Any build/config files?
   Makefile, Dockerfile, docker-compose.yml, tsconfig.json, webpack.config.*

If ANY of the above are found: existing codebase detected.
If NONE found: pure greenfield, skip brownfield flow silently.

### Constants
CODEBASE_MAP_PATH = '.planning/CODEBASE.md'
MAX_TREE_DEPTH = 2
MAX_FILE_SAMPLE = 10  (per category — don't enumerate every file)
STALE_THRESHOLD_DAYS = 30

### Graceful Degradation
- If CODEBASE.md does not exist: all consumers skip brownfield context silently
- If CODEBASE.md is stale (>30 days): consumers warn but do not block
- Never error, never block, never require brownfield analysis for workflow completion
```

**Section 2: Map Generation (BROWN-01)** — Building the structural map of the codebase:

```
### 2.1: File Tree Summary
Generate a depth-limited directory tree, excluding noise:
  Command (Bash):
    find . -maxdepth {MAX_TREE_DEPTH} \
      -not -path './.git/*' \
      -not -path './node_modules/*' \
      -not -path './.planning/*' \
      -not -path './.claude/*' \
      -not -name '*.lock' \
      -not -name '.DS_Store' \
      | sort

For large codebases (>500 entries at depth 2): reduce to depth 1 and summarize subdirectories by file count.

### 2.2: Language Distribution
Count files by extension:
  - Glob("**/*.{ext}") for each detected extension
  - Calculate percentage of total source files
  - Sort by count descending
  - Only include extensions with >= 2 files (filter noise)

Output format:
| Extension | File Count | % of Codebase |
|-----------|-----------|---------------|

### 2.3: Entry Points
Identify project entry points by checking for:
- package.json "main" / "module" / "bin" fields
- Python: __main__.py, manage.py, app.py, main.py
- Ruby: config.ru, bin/rails
- Go: main.go, cmd/ directory
- Generic: src/index.*, app/main.*, Makefile targets

### 2.4: Module Structure
Detect how the codebase is organized:
- Monorepo: multiple package.json / go.mod files in subdirectories
- Flat: all source in root or single src/ directory
- Domain-driven: directories named by business domain
- MVC: controllers/, models/, views/ directories
- Component-based: components/, modules/, features/ directories

Use directory name heuristics — do not read every file.
```

**Section 3: Pattern Detection (BROWN-02)** — Identifying frameworks, libraries, and conventions:

```
### 3.1: Framework Detection Protocol
Two-stage detection to avoid false positives:

Stage 1: Check indicator file exists (Glob)
Stage 2: Grep for specific framework marker within that file

#### Node.js / JavaScript Ecosystem
Indicator file: package.json
If exists, Read package.json and check dependencies/devDependencies:
  - "express" → Express.js
  - "fastify" → Fastify
  - "next" → Next.js
  - "react" → React
  - "vue" → Vue.js
  - "@angular/core" → Angular
  - "svelte" → Svelte
  - No framework deps → vanilla Node.js

Test frameworks (check devDependencies):
  - "jest" → Jest
  - "vitest" → Vitest
  - "mocha" → Mocha
  - "cypress" → Cypress
  - "playwright" → Playwright

#### Python Ecosystem
Indicator files: requirements.txt, pyproject.toml, setup.py, Pipfile
Read the indicator file and check for:
  - "django" → Django
  - "flask" → Flask
  - "fastapi" → FastAPI
  - "pytest" → pytest test suite

#### Ruby Ecosystem
Indicator file: Gemfile
Read Gemfile and check for:
  - gem 'rails' → Ruby on Rails
  - gem 'sinatra' → Sinatra
  - gem 'rspec' → RSpec test suite

#### Go Ecosystem
Indicator file: go.mod
Read go.mod and check for:
  - "github.com/gin-gonic/gin" → Gin
  - "github.com/labstack/echo" → Echo
  - "github.com/gorilla/mux" → Gorilla Mux

#### Rust Ecosystem
Indicator file: Cargo.toml
Read Cargo.toml and check for:
  - "actix-web" → Actix Web
  - "rocket" → Rocket
  - "tokio" → Tokio async runtime

#### Unknown / Custom
If none of the above match:
  - Report "Custom / Unknown" with file extensions as evidence
  - Do not guess — state the evidence and let the planner decide

### 3.2: Convention Detection
Analyze naming patterns and structure:
  - File naming: kebab-case, camelCase, snake_case, PascalCase
    (sample 10 source files and check naming pattern)
  - Test location: tests next to source, separate test/ directory, or __tests__/
  - Config style: .env files, config/ directory, environment-specific configs
  - Import style: relative vs absolute, barrel files (index.ts re-exports)
  - Linting/formatting: .eslintrc, .prettierrc, .rubocop.yml, pyproject.toml [tool.black]

### 3.3: Architecture Style Inference
Based on directory names and structure, infer:
  - MVC: controllers/ + models/ + views/
  - Clean Architecture: domain/ + application/ + infrastructure/
  - Feature-based: features/{name}/ with co-located files
  - Layered: api/ + services/ + repositories/
  - Flat: all files in root or single directory
  - Monorepo: packages/ or apps/ with individual package manifests

Note: inference is heuristic. Include "(inferred from directory structure)" in evidence.
```

**Section 4: Risk Assessment (BROWN-03)** — Flagging complexity, debt, and hotspots:

```
### 4.1: Complexity Indicators
Find large files (high complexity signal):
  Command (Bash):
    find . -name '*.{ext}' | xargs wc -l 2>/dev/null | sort -rn | head -20

Thresholds (per file):
  - >500 lines: HIGH complexity
  - 200-500 lines: MEDIUM complexity
  - <200 lines: LOW complexity

### 4.2: Technical Debt Markers
Count TODO/FIXME/HACK/XXX markers:
  Command (Grep):
    Pattern: "TODO|FIXME|HACK|XXX"
    Across all source files

Calculate per-file rate:
  total_markers / total_source_files = debt_density
  - > 1.0 markers/file: HIGH debt signal
  - 0.3-1.0 markers/file: MEDIUM debt signal
  - < 0.3 markers/file: LOW debt signal

### 4.3: Git Hotspot Detection (optional — only if git repo)
Files changed most in the last 90 days:
  Command (Bash):
    git log --since="90 days ago" --name-only --format="" | sort | uniq -c | sort -rn | head -10

If git is not available (not a repo, or no history): skip silently.

### 4.4: Dependency Risk
Check for warning signs:
  - Missing lockfile: package.json without package-lock.json or yarn.lock
  - Missing lockfile: Gemfile without Gemfile.lock
  - No .gitignore: risk of committed artifacts
  - No CI config: .github/workflows/, .gitlab-ci.yml, Jenkinsfile absent
  - Outdated dependency manifests: check if lockfile exists but is significantly older than manifest

### 4.5: Risk Summary
Produce a risk table:
| Area | Risk Level | Why | Recommendation |
|------|-----------|-----|----------------|

Risk levels are relative to project size:
  - Use per-file rates, not absolute counts
  - A 5-file project with 10 TODOs is HIGH; a 500-file project with 10 TODOs is LOW
  - Calibrate complexity thresholds to the project's average file size
```

- Conventions Detected: bullet list
- Entry Points: table
- Risk Areas: table with Area, Risk Level, Why, Recommendation
- Technical Debt Signals: bullet list with counts
