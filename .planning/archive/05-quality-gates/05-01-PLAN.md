Create `.claude/skills/agency/review-loop.md`. This skill is loaded by `/agency:review` and provides the complete methodology for running dev-QA review cycles with personality-injected testing agents.

Format:
```yaml
---
name: agency:review-loop
description: Dev-QA loop engine with structured feedback, fix routing, and user escalation for /agency:review
---
```

Structure the skill with these sections:

**Section 1: Review Principles**
Brief statement of core rules:
- Reviews evaluate the OUTPUT of a phase (the files created/modified during /agency:build), not the plans themselves
- Review agents are personality-injected: they receive their full .md file as system prompt
- Feedback must be structured and specific: file/artifact reference, severity, exact issue, suggested fix
- The loop is: review → collect findings → fix → re-review, max 3 cycles total
- Reviews use Sonnet for both review agents and fix agents (cost profile: execution = Sonnet)
- A phase is NOT marked complete until review agents approve it — no passing by exhaustion
- If 3 cycles fail to resolve all blockers, escalate to the user with full context
- Review agents should be skeptical by default — "no issues found" on first review is a yellow flag

**Section 2: Review Agent Selection**
How to choose the right review agents for a phase:

```
Step 1: Determine the phase type
  - Read the phase's CONTEXT.md to understand what was built
  - Read all SUMMARY.md files to see what artifacts were produced
  - Classify the phase into one or more types:
    - "code" — new code files, skills, commands, scripts
    - "api" — API endpoints, integrations, external service connections
    - "design" — UI components, design systems, visual assets
    - "marketing" — content, campaigns, strategy documents
    - "infrastructure" — CI/CD, deployment, configuration
    - "workflow" — process documents, methodology files, skill documents

Step 2: Select review agents based on phase type
  Phase type → Primary reviewer → Secondary reviewer:
  - code → testing-reality-checker → testing-evidence-collector
  - api → testing-api-tester → testing-reality-checker
  - design → design-brand-guardian → testing-reality-checker
  - marketing → testing-workflow-optimizer → testing-reality-checker
  - infrastructure → testing-reality-checker → testing-performance-benchmarker
  - workflow → testing-workflow-optimizer → testing-reality-checker

  Rules:
  - Always include testing-reality-checker (either primary or secondary) — they are the final quality gate
  - For phases touching multiple types, include one reviewer per type (max 3 reviewers total)
  - Present selected reviewers to the user for confirmation before spawning

Step 3: Validate reviewer availability
  - Check that the selected agent .md files exist at the expected paths
  - If a personality file is missing: fall back to testing-reality-checker for that slot
```

**Section 3: Review Prompt Construction**
How to build the prompt that review agents receive:

```
For each review agent:

Step 1: Gather phase artifacts
  - Read all PLAN.md files in the phase directory
  - Read all SUMMARY.md files in the phase directory
  - From each plan's files_modified list, build the full list of files to review
  - Read the phase CONTEXT.md for the phase goal and success criteria

Step 2: Read the reviewer's personality file
  - Path from agent-registry: agency-agents/{division}/{agent-id}.md
  - Read the ENTIRE file — do not truncate or summarize

Step 3: Construct the review prompt
  Format:
  """
  {full personality .md content}

  ---

  # Review Task

  You are reviewing the output of Phase {N}: {phase_name} for The Agency Workflows.

  ## Phase Goal
  {goal from CONTEXT.md}

  ## Success Criteria (from ROADMAP.md)
  {bulleted success criteria for this phase}

  ## Plans Executed
  {For each plan: plan number, name, status from SUMMARY.md}

  ## Files to Review
  {Complete list of files created/modified, from plan frontmatter files_modified}

  ## Your Review Instructions

  1. Read EVERY file in the "Files to Review" list above
  2. For each file, evaluate against the success criteria and the plan's verification checklist
  3. Check for:
     - Correctness: Does the code/content do what the plan specified?
     - Completeness: Are all tasks from the plan actually implemented?
     - Consistency: Does it follow established patterns from existing skills/commands?
     - Integration: Do references between files resolve correctly (e.g., @-references, path imports)?
     - Quality: Is the code/content at a reasonable quality level?
  4. Report findings in the EXACT format below — no other format is accepted

  ## Required Feedback Format

  For EACH finding, use this exact structure:

  ### Finding {N}
  - **File**: {file path}
  - **Line/Section**: {line number or section name}
  - **Severity**: {BLOCKER | WARNING | SUGGESTION}
  - **Issue**: {One sentence describing the exact problem}
  - **Details**: {2-3 sentences explaining why this is a problem and what should be true instead}
  - **Suggested Fix**: {Specific action to resolve — not vague guidance}

  Severity definitions:
  - BLOCKER: Prevents the phase from working correctly. Must be fixed before approval.
  - WARNING: Works but has a meaningful quality/consistency issue. Should be fixed.
  - SUGGESTION: Minor improvement opportunity. Nice to have, not required.

  ## Final Verdict

  After all findings, provide exactly ONE of:
  - **PASS** — No blockers, no warnings (or warnings already addressed). Phase is approved.
  - **NEEDS WORK** — Has blockers or significant warnings. List the specific findings that must be addressed.
  - **FAIL** — Fundamental issues requiring substantial rework. Explain what went wrong at a structural level.

  IMPORTANT:
  - Do NOT give letter grades, numeric scores, or vague assessments like "looks good"
  - Default to NEEDS WORK — first reviews almost always find issues
  - Every finding MUST reference a specific file and location
  - "No issues found" requires a brief explanation of what you checked and why you're confident
  - Compare what was BUILT against what was SPECIFIED — not against hypothetical perfection
  """

Step 4: Spawn the review agent
  Agent tool parameters:
  - subagent_type: "general-purpose"
  - prompt: {constructed prompt from Step 3}
  - model: "sonnet"
  - name: "{agent-id}-review-{NN}" (e.g., "testing-reality-checker-review-05")
  - team_name: "{phase_team_name}"
```

**Section 4: Feedback Collection and Triage**
How to process review agent feedback:

```
After each review agent completes and sends its findings via SendMessage:

Step 1: Parse findings
  - Extract each Finding block from the agent's response
  - Categorize by severity: BLOCKER, WARNING, SUGGESTION
  - Record the file and location for each finding

Step 2: Deduplicate across reviewers
  - If multiple reviewers flag the same file+location: keep the highest severity
  - If reviewers disagree on the same issue: escalate to BLOCKER

Step 3: Triage
  - Must-fix list: all BLOCKERs + WARNINGs
  - Nice-to-have list: all SUGGESTIONs
  - If the must-fix list is empty and all reviewers gave PASS: review passes, go to Section 7
  - If the must-fix list has items: proceed to Section 5 (Fix Cycle)

Step 4: Report findings to user
  Display a summary table:
  ## Review Findings — Cycle {cycle_number}/3

  | # | Severity | File | Issue | Reviewer |
  |---|----------|------|-------|----------|
  | 1 | BLOCKER  | path | brief | agent-id |
  | 2 | WARNING  | path | brief | agent-id |
  ...

  Blockers: {count} | Warnings: {count} | Suggestions: {count}
  Verdict: {PASS | NEEDS WORK | FAIL}
```

**Section 5: Fix Cycle**
How to route fixes and iterate:

```
For each must-fix finding:

Step 1: Determine the fix agent
  - Use agent-registry Section 3 (Recommendation Algorithm) to match the finding's file type to an agent:
    - .md skill/command files → autonomous (no personality needed — these are methodology docs)
    - .md agent personality files → autonomous
    - Code files (.js, .ts, .py, etc.) → match to relevant engineering agent
    - Design assets → match to relevant design agent
    - Marketing content → match to relevant marketing agent
  - If no clear match: use autonomous mode

Step 2: Construct the fix prompt
  Format:
  """
  {personality content if not autonomous, otherwise skip}

  ---

  # Fix Task

  You are fixing issues found during a quality review of Phase {N}: {phase_name}.

  ## Findings to Fix
  {Include only the findings assigned to this agent}

  For each finding:
  - Read the referenced file
  - Apply the suggested fix (or a better fix if you have specialist insight)
  - Verify the fix resolves the issue
  - Do NOT introduce new issues
  - Do NOT modify files beyond what is needed to fix the listed findings

  ## Report
  After fixing, report:
  - Which findings you fixed (by number)
  - What you changed (file, line, before → after)
  - Any findings you could NOT fix and why
  """

Step 3: Spawn fix agents
  - Group findings by fix agent to minimize agent spawns
  - Spawn fix agents in parallel (same pattern as wave-executor Section 4)
  - model: "sonnet" for fix agents
  - name: "{agent-id}-fix-{NN}-cycle{C}" (e.g., "executor-fix-05-cycle1")

Step 4: Collect fix results
  - Wait for all fix agents to complete via SendMessage
  - Record which findings were fixed
  - Create an atomic git commit for the fixes:
    fix(agency): review cycle {C} fixes for phase {N}

    Fixed {count} issues: {brief list}

    Co-Authored-By: Claude Opus 4.6 <noreply@anthropic.com>
```

**Section 6: Re-review Cycle**
How to iterate the review → fix → re-review loop:

```
After fix agents complete:

Step 1: Increment cycle counter
  - cycle_count += 1
  - If cycle_count > 3: go to Section 8 (Escalation)

Step 2: Determine what to re-review
  - Only re-review files that were modified by fix agents
  - Plus any files referenced by findings that were NOT fixed
  - Do NOT re-review the entire phase — scope the re-review to changed areas

Step 3: Spawn review agents for re-review
  - Use the SAME review agents from the initial review (consistency)
  - Modify the review prompt to include:
    - Previous findings and their fix status
    - Instruction to verify fixes AND check for regressions
    - Instruction to focus on previously-flagged areas

Step 4: Process re-review results
  - Same process as Section 4 (Feedback Collection)
  - If new BLOCKERs or WARNINGs found: go back to Section 5 (Fix Cycle)
  - If all findings resolved: review passes, go to Section 7

Step 5: Track cycle progress
  Update STATE.md after each cycle:
  - Status: "Phase {N} under review — cycle {C}/3, {blocker_count} blockers remaining"
```

**Section 7: Review Passed**
What happens when all reviewers approve:

```
When review passes (no remaining BLOCKERs or WARNINGs):

Step 1: Generate review summary
  Write .planning/phases/{NN}-{slug}/{NN}-REVIEW.md:
  # Phase {N}: {phase_name} — Review Summary

  ## Result: PASSED
  **Cycles**: {total_cycles_used}
  **Reviewers**: {list of reviewer agent IDs}

  ## Findings Summary
  - Total findings: {count}
  - Blockers fixed: {count}
  - Warnings fixed: {count}
  - Suggestions (not required): {count}

  ## Findings Detail
  {For each finding: number, severity, file, issue, resolution, cycle fixed}

  ## Reviewer Verdicts
  {For each reviewer: agent ID, final verdict, key observations}

Step 2: Mark phase complete
  Follow execution-tracker Section 4 (Phase Completion Tracking):
  - Update STATE.md:
    - Phase: {N} of {total} (complete)
    - Status: "Phase {N} complete — review passed ({cycles} cycle(s))"
    - Last Activity: "Phase {N} review ({date})"
    - Next Action: "Run `/agency:plan {N+1}` for the next phase" or "Project complete!" if last phase
  - Update ROADMAP.md:
    - Check the phase row: [x]
    - Status: "Complete"

Step 3: Create review completion commit
  chore(agency): phase {N} review passed — {phase_name}

  Review passed after {cycles} cycle(s).
  {blocker_count} blockers fixed, {warning_count} warnings fixed.

  Co-Authored-By: Claude Opus 4.6 <noreply@anthropic.com>

Step 4: Route to next action
  - If there are more phases: "Phase {N} complete. Run `/agency:plan {N+1}` for the next phase."
  - If this was the last phase: "All phases complete! Project review finished."
```

**Section 8: Escalation**
What happens when 3 cycles are exhausted without resolution:

```
When cycle_count exceeds 3 and BLOCKERs remain:

Step 1: Generate escalation report
  Write .planning/phases/{NN}-{slug}/{NN}-REVIEW.md with:
  # Phase {N}: {phase_name} — Review Summary

  ## Result: ESCALATED
  **Cycles Used**: 3 (maximum reached)
  **Remaining Blockers**: {count}

  ## Unresolved Findings
  {For each unresolved BLOCKER: full finding detail, what was attempted, why it wasn't resolved}

  ## Resolved Findings
  {For each finding that WAS fixed: finding detail, resolution, cycle fixed}

  ## Recommendation
  {Brief assessment: are the remaining issues fundamental (rework needed) or targeted (specific fixes needed)?}

Step 2: Update STATE.md
  - Status: "Phase {N} review escalated — {count} unresolved blocker(s)"
  - Next Action: "Review .planning/phases/{NN}-{slug}/{NN}-REVIEW.md and decide: fix manually, re-run /agency:review, or accept as-is"

Step 3: Present to user
  ## Phase {N}: {phase_name} — Review Escalated

  3 review cycles completed. {count} blocker(s) remain unresolved.

  | # | File | Issue | Attempted Fix |
  |---|------|-------|---------------|
  ...

  Options:
  - Fix the remaining issues manually, then re-run `/agency:review`
  - Accept the phase as-is and proceed to `/agency:plan {N+1}`
  - Investigate the root cause before proceeding

  Do NOT auto-proceed. Wait for user decision.
```
