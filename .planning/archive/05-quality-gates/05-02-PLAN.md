**process** — Fully specified steps (not placeholders):

```
1. DETERMINE TARGET PHASE
   - Check $ARGUMENTS for --phase N flag (e.g., `/agency:review --phase 4`)
   - If no flag: read STATE.md to determine current phase
     - Use the phase number from "Phase: N of M" in Current Position
     - Valid states for review: "executed, pending review" or "partial"
     - If status says "planned" or "pending": error — "Phase {N} hasn't been executed yet. Run /agency:build first."
     - If status says "complete": error — "Phase {N} already passed review. Run /agency:plan {N+1} for the next phase."
   - Validate: phase must exist in ROADMAP.md
   - Check that phase directory has SUMMARY.md files (proof of execution):
     - If no SUMMARY.md files: error — "Phase {N} has no execution summaries. Run /agency:build first."
   - Display: "Reviewing Phase {N}: {phase_name}"

2. GATHER PHASE CONTEXT
   Follow review-loop skill Section 3, Step 1 (Gather phase artifacts):
   - Read the phase CONTEXT.md for the phase goal and success criteria
   - Read all PLAN.md files in the phase directory to understand what should have been built
   - Read all SUMMARY.md files to understand what was actually done
   - From each plan's files_modified, build the complete list of files to review
   - Read the ROADMAP.md success criteria for this phase
   - Display context summary:
     "Phase {N}: {phase_name}
      Goal: {phase_goal}
      Plans executed: {count} ({list names and statuses})
      Files to review: {count}
      {file list}"

3. SELECT REVIEW AGENTS
   Follow review-loop skill Section 2 (Review Agent Selection):
   a. Classify the phase type from its CONTEXT.md and SUMMARY.md content
      (code, api, design, marketing, infrastructure, workflow)
   b. Map phase type to primary + secondary review agents using the selection table
   c. Present selected reviewers to user via AskUserQuestion:
      "Review agents for Phase {N}: {phase_name}?"
      Options:
      - "{primary_agent_name} + {secondary_agent_name}" (Recommended)
      - "{primary_agent_name} only" — single reviewer, faster but less thorough
      - "{alternative_agent_name} + {primary_agent_name}" — different reviewer pair
   d. If user selects "Other": accept custom agent IDs from user input
   e. Validate selected agents exist in agent-registry

4. EXECUTE REVIEW CYCLE
   Initialize: cycle_count = 0

   **Team Setup** (once, before review loop):
   - Call TeamCreate with team_name: "phase-{NN}-review" (e.g., "phase-05-review")

   **LOOP START** (max 3 iterations):

   a. Increment cycle_count
      Announce: "Review cycle {cycle_count}/3"

   b. Spawn review agents (review-loop Section 3):
      For each selected reviewer:
      - Read the reviewer's personality .md file (full content)
      - Read all files in the review file list
      - Construct the review prompt per review-loop Section 3, Step 3:
        personality + phase context + success criteria + review instructions + feedback format
      - Spawn via Agent tool:
        - subagent_type: "general-purpose"
        - model: "sonnet"
        - name: "{agent-id}-review-{NN}-c{cycle}" (e.g., "testing-reality-checker-review-05-c1")
        - team_name: "phase-{NN}-review"
      - Spawn ALL reviewers in parallel (multiple Agent calls in one message)

   c. Collect review results via SendMessage (review-loop Section 4):
      - Wait for all review agents to send findings
      - Parse findings into structured format: file, line/section, severity, issue, details, fix
      - Deduplicate across reviewers (highest severity wins on conflicts)
      - Triage: must-fix (BLOCKER + WARNING) vs nice-to-have (SUGGESTION)

   d. Display findings table to user (review-loop Section 4, Step 4):
      ## Review Findings — Cycle {cycle_count}/3
      | # | Severity | File | Issue | Reviewer |
      ...
      Blockers: {count} | Warnings: {count} | Suggestions: {count}
      Verdict: {aggregate verdict}

   e. If verdict is PASS (no blockers, no warnings):
      - Break the loop — go to step 5
   f. If verdict is NEEDS WORK or FAIL AND cycle_count < 3:
      - Route fixes per review-loop Section 5 (Fix Cycle):
        - Determine fix agent for each finding (agent-registry or autonomous)
        - Group findings by fix agent
        - Construct fix prompts per review-loop Section 5, Step 2
        - Spawn fix agents in parallel
        - Wait for fix results via SendMessage
        - Create fix commit:
          fix(agency): review cycle {C} fixes for phase {N}
        - Go back to LOOP START for re-review
          (re-review scopes to modified files per review-loop Section 6)
   g. If cycle_count >= 3 AND blockers remain:
      - Break the loop — go to step 5 (escalation path)

   **LOOP END**

   **Team Teardown** (always, after loop ends):
   - Send shutdown_request to all agents via SendMessage
   - Wait for shutdown confirmations
   - Call TeamDelete to clean up

5. COMPLETE REVIEW
   Determine outcome:

   **Path A: Review Passed** (review-loop Section 7)
   - Generate .planning/phases/{NN}-{slug}/{NN}-REVIEW.md with review summary
   - Mark phase complete in STATE.md and ROADMAP.md:
     - STATE.md Phase: {N} of {total} (complete)
     - STATE.md Status: "Phase {N} complete — review passed ({cycles} cycle(s))"
     - STATE.md Next Action: "Run `/agency:plan {N+1}` for the next phase"
     - ROADMAP.md: check [x], Status = "Complete"
   - Update REQUIREMENTS.md: mark QA requirements as complete
   - Create review completion commit:
     chore(agency): phase {N} review passed — {phase_name}
   - Display:
     "Phase {N}: {phase_name} — Review PASSED ({cycles} cycle(s))
      Run `/agency:plan {N+1}` for the next phase."

   **Path B: Review Escalated** (review-loop Section 8)
   - Generate .planning/phases/{NN}-{slug}/{NN}-REVIEW.md with escalation report
   - Update STATE.md:
     - Status: "Phase {N} review escalated — {count} unresolved blocker(s)"
     - Next Action: "Review {NN}-REVIEW.md and decide next steps"
   - Display escalation table with remaining blockers
   - Use AskUserQuestion: "How would you like to proceed?"
     Options:
     - "Fix manually and re-run /agency:review" — exit, let user work
     - "Accept as-is and move to /agency:plan {N+1}" — mark phase complete despite issues
     - "Investigate further" — exit for user to diagnose
   - If user accepts as-is:
     - Mark phase complete (same as Path A)
     - Note in REVIEW.md: "Accepted with {count} unresolved blocker(s) by user decision"

6. ROUTE TO NEXT ACTION
   - If review passed or user accepted:
     "Phase {N}: {phase_name} complete.
      Next: Run `/agency:plan {N+1}` to plan the next phase."
     OR if this was the last phase:
     "All phases complete! The Agency Workflows project is finished."
   - If escalated and user chose to fix manually:
     "Fix the issues listed in .planning/phases/{NN}-{slug}/{NN}-REVIEW.md
      Then re-run `/agency:review` to verify."
   - Do NOT automatically trigger /agency:plan — let the user decide when.
```

- The command handles all three outcomes: pass, escalation, and user override
