# Phase 3: Phase Planning — Context

## Phase Goal
Users can type `/agency:plan 1` and get wave-structured plans with recommended agents for that phase.

## Requirements Covered
- PLAN-01: `/agency:plan` command — plans a specific phase
- PLAN-02: Agent recommendation skill — analyzes phase tasks and recommends agents from the 51
- PLAN-03: User confirmation gate — present recommended agents, allow override
- PLAN-04: Phase decomposition into wave-structured plans (max 3 tasks per plan)
- PLAN-05: Plan files written to `.planning/phases/{N}/`

### Plan File Format (established in Phases 1-2)
The plan format from existing `.planning/phases/` plans:
```yaml
---
phase: {NN}-{phase-name}
plan: {PP}
type: execute
wave: {N}
depends_on: ["{NN}-{PP}", ...]  # plans that must complete first
files_modified: [...]
autonomous: true|false
requirements: [REQ-XX, ...]
user_setup: []
must_haves:
  truths: [...]       # invariants that must hold
  artifacts: [...]    # files produced with min_lines, contains checks
  key_links: [...]    # from/to/via/pattern dependency links
---

<objective>...</objective>
<context>@references to state and source files</context>
<tasks>
  <task type="auto|user">
    <name>...</name>
    <files>...</files>
    <action>detailed instructions</action>
    <verify>bash verification commands</verify>
    <done>completion sentence</done>
  </task>
</tasks>
<verification>checklist of all conditions</verification>
<success_criteria>bulleted outcomes</success_criteria>
<output>summary file + state updates</output>
```

## Key Design Decisions

### Phase-decomposer is a skill, not an agent
Like questioning-flow, the phase-decomposer tells Claude HOW to decompose a phase — it's loaded as instructions, not spawned as a separate agent. This keeps `/agency:plan` as a single-context operation with full visibility.

### Agent recommendation is per-plan, not per-phase
The ROADMAP.md assigns agents at the phase level, but during `/agency:plan`, agents are refined and assigned per-plan. A phase with 2 plans may use different agents for each plan (e.g., frontend engineer for Plan 1, backend architect for Plan 2).

### User confirms agents before plans are written
The flow is: analyze → recommend → present → confirm → write plans. The user sees which agents will work on what BEFORE the plan files are generated. This matches the "hybrid selection" pattern.

### AskUserQuestion for structured agent selection
Agent confirmation uses AskUserQuestion with the recommended agents as options, plus an "Other" escape hatch. This is consistent with /agency:start's preference collection pattern.

### Max 3 tasks per plan, max 3 plans per phase
Each plan has at most 3 tasks. Most phases should decompose into 2-3 plans organized by dependency wave. This keeps individual plans focused and reviewable.

### Plan auto-detection
If the user runs `/agency:plan` without a phase number, the command should detect the next unplanned phase from ROADMAP.md (first phase where Status = Pending).
