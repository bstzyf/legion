Structure the skill with these sections:

**Section 1: Decomposition Principles**
Brief statement of core rules:
- Every plan has max 3 tasks — if more work is needed, create more plans
- Plans are organized into waves — wave 1 has no internal dependencies, wave 2 depends on wave 1 outputs, etc.
- Agent recommendations are per-plan — different plans in the same phase may use different agents
- Plans must be self-contained enough for an agent to execute with only its context references
- Each task must have a concrete verify step — no "manually check" instructions
- Prefer fewer plans over more — 2 focused plans beats 4 thin ones

**Section 2: Phase Analysis**

How to extract and analyze phase information from ROADMAP.md:

```
Step 1: Read .planning/ROADMAP.md
Step 2: Find the Phase Details section for the target phase number
Step 3: Extract:
  - phase_name: the descriptive name (e.g., "Plugin Foundation")
  - phase_goal: the Goal field
  - phase_requirements: the Requirements field (list of requirement IDs)
  - success_criteria: the Success Criteria list
  - estimated_plans: the Plans count
Step 4: Read .planning/REQUIREMENTS.md
Step 5: Cross-reference: for each requirement ID in phase_requirements,
        get the full requirement description from REQUIREMENTS.md
Step 6: Read .planning/PROJECT.md for broader project context
Step 7: Read .planning/STATE.md for current progress and any completed phases
        that this phase builds on
```

**Section 3: Task Decomposition**

How to break phase work into plans and tasks:

```
1. LIST all concrete deliverables implied by the phase requirements
   - Each requirement may produce 1-3 deliverables (files, features, configs)
   - Group deliverables by dependency: which ones need others to exist first?

2. IDENTIFY dependency layers
   - Layer 0: Deliverables with no dependencies (foundations, new files)
   - Layer 1: Deliverables that depend on Layer 0 outputs
   - Layer 2: Deliverables that depend on Layer 1 outputs
   - Rarely more than 3 layers needed

3. MAP layers to waves
   - Each dependency layer becomes a wave
   - Within a wave, work can happen in parallel (no internal dependencies)

4. GROUP deliverables into plans within each wave
   - Max 3 tasks per plan
   - Group by: same file being modified, same skill/pattern, same agent specialty
   - Name plans by their primary output: "Create {X} skill", "Update {Y} command"

5. VALIDATE plan structure
   - Every requirement from ROADMAP.md is covered by at least one task
   - No circular dependencies between waves
   - Each plan can be executed independently of other plans in the same wave
   - Total plan count matches or is close to ROADMAP.md estimate
```

**Section 4: Agent Recommendation**

How to select agents for each plan using agent-registry:

```
For each plan:

1. ANALYZE the plan's tasks
   - What type of work? (new skill, command update, test writing, etc.)
   - What domains are involved? (frontend, backend, design, content, etc.)
   - What files are being created/modified?

2. MATCH against agent-registry
   - Use Section 3 (Recommendation Algorithm) of agent-registry.md
   - Extract task type terms from plan tasks
   - Score agents by: exact match (3pts), partial match (1pt), division alignment (2pts)
   - Apply mandatory roles: testing agent for code work, coordinator for cross-division

3. SELECT recommended agents
   - Pick top 1-2 agents per plan (not per task — agents work at plan level)
   - For skills/markdown-only plans: may not need agents (autonomous execution)
   - For code-heavy plans: must include at least 1 engineering + 1 testing agent

4. FORMAT recommendation for user presentation
   For each plan:
   - Plan name and wave
   - Recommended agent(s) with ID, division, and 1-sentence rationale
   - What the agent will do (brief)
   - Whether the plan is autonomous (Claude executes directly) or agent-delegated
```

**Section 7: Context File Generation**

Before writing plan files, generate the phase context file:

```
File: .planning/phases/{NN}-{phase-slug}/{NN}-CONTEXT.md

Content:
- Phase Goal (from ROADMAP.md)
- Requirements Covered (from ROADMAP.md + REQUIREMENTS.md)
- What Already Exists (files/skills/commands from prior phases)
- Key Design Decisions (rationale for decomposition choices)
- Plan Structure (list of plans with wave assignments)
```

The context file is written FIRST, then referenced by all plan files.

**Section 8: Edge Cases**

- **Phase already planned**: Check if `.planning/phases/{NN}-{slug}/` already has PLAN.md files. If so, warn the user and confirm before overwriting.
- **No phase number given**: Auto-detect the next unplanned phase — first phase in ROADMAP.md where the progress table shows 0 completed plans and no existing plan files.
- **Phase doesn't exist**: If the given phase number exceeds the roadmap's phase count, error with: "Phase {N} doesn't exist in ROADMAP.md. The roadmap has {count} phases."
- **Phase already complete**: If the progress table shows the phase as Complete, warn: "Phase {N} is already marked complete. Re-plan anyway?" with AskUserQuestion.
- **Single-task phase**: Some phases may only need 1 plan with 1-2 tasks. That's fine — don't pad to hit a plan count.
- **Cross-division phase**: If a phase spans multiple divisions, ensure at least one plan includes a coordinator agent (project-manager-senior, project-shepherd, or agents-orchestrator).
