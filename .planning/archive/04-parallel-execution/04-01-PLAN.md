```yaml
name: agency:wave-executor
description: Executes wave-structured plans with parallel personality-injected agents via Claude Code Teams

```

Structure the skill with these sections:

**Section 1: Execution Principles**
Brief statement of core rules:
- Plans are executed wave-by-wave: all Wave 1 plans complete before Wave 2 starts
- Within a wave, all plans execute in parallel via Claude Code Teams
- Each agent receives its FULL personality .md file as system prompt — no summaries, no excerpts
- Agents use Sonnet model (cost profile: execution = Sonnet)
- Failed agents do NOT retry automatically — errors are captured and reported
- Each plan produces a summary file on completion
- The orchestrator (build command) stays in the main context; agents get fresh contexts

**Section 2: Plan Discovery**
How to find and organize plans for execution:

```
Step 1: Determine the phase directory path
  - From STATE.md or --phase argument, get the phase number
  - Construct path: .planning/phases/{NN}-{phase-slug}/
  - Read directory contents

Step 2: Find all plan files
  - Pattern: {NN}-{PP}-PLAN.md
  - Exclude: CONTEXT.md, SUMMARY.md files
  - Sort by plan number (PP)

Step 3: Read each plan's YAML frontmatter
  - Extract: wave, depends_on, autonomous, files_modified, requirements
  - Build a wave map: { wave_number: [plan_list] }

Step 4: Validate plan structure
  - All depends_on references must point to plans in earlier waves
  - No circular dependencies
  - At least one plan must exist
  - If validation fails: report error, do not proceed

Step 5: Determine execution order
  - Sort waves numerically (1, 2, 3, ...)
  - Within each wave, plans are unordered (parallel)
```

**Section 3: Personality Injection**
How to load and inject an agent personality for plan execution:

```
For each plan that has an agent assigned (autonomous: false):

Step 1: Read the agent personality file
  - Path from agent-registry: agency-agents/{division}/{agent-id}.md
  - Read the ENTIRE file — do not truncate or summarize

Step 2: Read the plan file
  - Read the full plan content (objective, context, tasks, verification)

Step 3: Construct the agent prompt
  Format:
  """
  {full personality .md content}

  ---

  # Execution Task

  You are executing a plan as part of The Agency Workflows. Follow the tasks below precisely.

  {full plan content starting from <objective>}

  ## Important
  - Execute each task in order
  - Run the <verify> commands after each task to confirm completion
  - After all tasks complete, run the <verification> checklist
  - Create a summary of what you did (files created/modified, key decisions, verification results)
  - Do NOT modify files outside of the plan's files_modified list unless absolutely necessary
  """

Step 4: Spawn the agent
  Agent tool parameters:
  - subagent_type: "general-purpose"
  - prompt: {constructed prompt from Step 3}
  - model: "sonnet"
  - name: "{agent-id}-{plan-number}" (e.g., "engineering-senior-developer-04-01")

For autonomous plans (autonomous: true):
  - Skip personality loading
  - Construct prompt with just the plan content
  - Use the same Agent tool call pattern but without personality prefix
  - Name: "executor-{plan-number}" (e.g., "executor-04-01")
```

**Section 4: Wave Execution**
How to execute plans wave by wave with parallel dispatch:

```
For each wave (in ascending order):

Step 1: Identify plans in this wave
  - Filter the wave map for current wave number
  - List plans to execute

Step 2: Pre-execution check
  - Verify all depends_on plans from previous waves have completed
  - If any dependency is missing/failed: skip this wave, report error

Step 3: Spawn agents in parallel
  - For SINGLE plan in wave: spawn directly via Agent tool (no Team needed)
  - For MULTIPLE plans in wave: use parallel Agent tool calls
    - Make all Agent tool calls in a SINGLE message to maximize parallelism
    - Each agent gets its own fresh context with personality + plan
  - All agents within the same wave are independent — no inter-agent communication needed

Step 4: Collect results
  - Each agent returns a result when complete
  - Parse the result for: success/failure, files modified, summary content
  - If an agent fails: capture the error output, mark the plan as failed

Step 5: Post-wave checkpoint
  - Report wave completion status to the orchestrator
  - List: plans completed, plans failed, files modified
  - If any plan failed: STOP execution, do not proceed to next wave
  - If all plans passed: proceed to next wave
```

**Section 5: Agent Result Processing**
How to handle what agents return:

```
For each completed agent:

Step 1: Parse the agent's return message
  - Look for: files created/modified, verification results, errors encountered
  - Extract the summary content

Step 2: Generate the plan summary file
  - Path: .planning/phases/{NN}-{slug}/{NN}-{PP}-SUMMARY.md
  - Format:
    # Plan {NN}-{PP} Summary: {plan_name}

    ## Result
    **Status**: Complete | Failed
    **Wave**: {wave_number}

    ## What Was Done
    {summary from agent}

    ## Files Created/Modified
    - {list}

    ## Key Details
    {notable decisions, implementation details}

    ## Verification Results
    {verification outcomes}

    ## Requirements Covered
    - {list from plan frontmatter}

Step 3: Handle failures
  - If the agent reported errors:
    - Write the summary with Status: Failed and error details
    - Include the full error output for diagnosis
    - Do NOT attempt to fix — this is for /agency:review
```

**Section 6: Error Scenarios**
How to handle common failure modes:

```
1. Agent spawn failure
   - The Agent tool itself fails (e.g., model unavailable)
   - Action: Mark plan as failed, capture error, continue with other plans in wave if any

2. Agent timeout
   - Agent takes too long (exceeds context limits)
   - Action: Capture whatever output exists, mark as failed

3. Verification failure
   - Agent completes tasks but verification steps fail
   - Action: Include verification output in summary, mark as complete-with-warnings

4. File conflict
   - Two agents in the same wave try to modify the same file
   - Prevention: Plan decomposition should prevent this (plans in same wave have no shared files)
   - If it happens: The second agent's changes may overwrite the first
   - Action: Flag the conflict in the wave completion report

5. Missing personality file
   - Agent .md file doesn't exist at the expected path
   - Action: Fall back to autonomous execution (no personality), log a warning

6. No plans found
   - Phase directory exists but has no PLAN.md files
   - Action: Error with "No plans found for Phase {N}. Run /agency:plan {N} first."
```
