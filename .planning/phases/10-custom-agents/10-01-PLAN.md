Create `.claude/skills/agency/agent-creator.md` -- a new skill file following the established pattern (YAML frontmatter, intro, numbered sections). Target 300-400 lines.

**Frontmatter:**
```yaml
---
name: agency:agent-creator
description: Guided agent personality creation with schema validation and registry integration
---
```

**Intro paragraph:** Reference workflow-common.md (personality paths, state locations), agent-registry.md (catalog format, recommendation algorithm), and questioning-flow.md (conversation pattern).

**Section 1: Agent Schema** -- Define the complete agent file schema:
- Required frontmatter fields with validation rules:
  - `name`: non-empty, kebab-case only (regex: `^[a-z][a-z0-9-]+$`), no spaces/uppercase/underscores, must match filename, must be globally unique (grep check)
  - `description`: non-empty, single line, at least 10 characters
  - `color`: must be one of: red, green, blue, purple, cyan, orange, yellow, pink
- Optional frontmatter fields:
  - `tools`: comma-separated list (when agent needs WebFetch, WebSearch, etc.)
- Required body sections (minimum viable agent):
  - `# {AgentName} Agent Personality` heading
  - Identity block (who the agent is, personality, experience)
  - Core mission (what the agent does, 3-5 capabilities)
  - Critical rules (hard rules the agent always follows)
  - Success criteria (how the agent knows it succeeded)
- Minimum 50 lines of body content
- Division placement: must be an existing division OR "custom" (new directory)

**Section 2: Guided Creation Flow** -- 3-stage adaptive conversation:

Stage 1 - Agent Identity (1-2 exchanges):
- Open with: "What kind of specialist do you want to add to your team? Give me the one-liner -- what does this agent do?"
- Extract: specialty domain, primary capability, differentiator from existing agents
- Infer: division (engineering/design/marketing/product/project-management/testing/support/spatial-computing/specialized/custom), suggested agent name (kebab-case)
- Confirm via AskUserQuestion: "I'll create a {division} agent -- '{suggested-name}'. Specialty: {description}. Correct, or would you like to adjust?"

Stage 2 - Capabilities and Personality (2-3 exchanges):
- Ask: "What are the top 3-5 things this agent can do that others can't?"
- Ask: "How does this agent think and communicate? What's its personality?"
- Ask: "Are there any hard rules it always follows?"
- Capture: capability list, personality traits, critical rules
- Confirm summary via AskUserQuestion before proceeding

Stage 3 - Registry Integration (1 exchange):
- Present generated task type tags (3-5 tags, kebab-case)
- Show existing registry tags from agent-registry.md as examples for alignment
- Use AskUserQuestion: "For recommendations to work, I need task type tags. Based on capabilities, I suggest: {tags}. These match existing registry tags. Add or remove any?"
- Confirm tags, then generate and present both the agent file content and the registry row for final confirmation

**Section 3: Schema Validation** -- Validation checklist run BEFORE writing any files:
1. Name uniqueness: `grep -rl "^name: {proposed_name}" agency-agents/` -- if output is non-empty, name is taken, prompt for alternative
2. Name format: must match `^[a-z][a-z0-9-]+$`
3. Description: non-empty, single line, at least 10 characters
4. Color: must be in allowed set
5. Division: must be valid existing division name or "custom"
6. Body content: at least 50 lines after frontmatter
7. Heading check: body must contain at least one `#` or `##` heading
8. Agent name must appear in the body content

If any validation fails: present specific errors via AskUserQuestion, allow correction, re-validate. Do NOT write files until all validations pass.

**Section 4: File Generation** -- After validation passes:
1. Construct agent file path: `agency-agents/{division}/{agent-name}.md`
2. If division is "custom" and `agency-agents/custom/` doesn't exist, create it
3. Write the agent .md file with:
   - YAML frontmatter (name, description, color, optionally tools)
   - Rich body content generated from conversation:
     - `# {AgentName} Agent Personality` heading
     - `## Your Identity` block from Stage 2 personality + domain
     - `## Core Mission` from Stage 2 capabilities (written as prose, not bullet-dump)
     - `## Critical Rules` from Stage 2 hard rules
     - `## Your Process` -- inferred workflow from capabilities
     - `## Success Criteria` -- how the agent measures its own success
   - Body must be substantive prose (~80-120 lines), NOT placeholder templates
4. Verify file exists after write: `test -f agency-agents/{division}/{agent-name}.md`

**Section 5: Registry Update** -- After file generation:
1. Read agent-registry.md
2. Find the target division table heading (e.g., `### Engineering Division`)
3. If division is "custom": create a new `### Custom Division` table section after the last existing division in Section 1
4. Insert a new row in the division table matching the exact format:
   `| {agent-id} | agency-agents/{division}/{agent-id}.md | {specialty description} | {tag1}, {tag2}, {tag3} |`
5. Update the division agent count in the heading (e.g., `### Engineering Division (8 agents)`)
6. For new "custom" division, also update the Section 1 intro text to mention the new division

**Section 6: Error Handling** -- Cover these scenarios:
- e1: User cancels during any stage -- exit gracefully, no partial files written
- e2: Name collision detected -- present conflict, suggest alternatives (append number or qualifier)
- e3: Division directory doesn't exist (for "custom") -- create it with `mkdir -p`
- e4: Agent file write fails -- report error, do not update registry (no partial state)
- e5: Registry update fails -- report error, the agent file is still valid and can be manually registered

**Section 7: Confirmation and Commit** -- After both file and registry are updated:
- Display summary: agent name, file path, division, task types, registry row
- Suggest: "Your new agent '{agent-name}' is ready. It will appear in `/agency:plan` recommendations for tasks matching: {tags}."
- Do NOT automatically commit -- the calling command handles that
