## Task 1: Add Compare Mode Workflow to Polymath Agent

**Action**: Extend `agents/polymath.md` with the compare mode workflow for structured alternative evaluation.

1. Add compare mode workflow phases (alongside crystallize and onboard phases):
   - **Phase 1 — Alternative Identification**: Identify 2-4 alternatives to compare.
     - If user provided alternatives: research each to understand capabilities, trade-offs
     - If user provided a problem: research and identify 2-4 candidate solutions
     - Present alternatives using structured choice for user confirmation:
       ```
       "I found these alternatives to compare. Which should we include?"
       → All of these (proceed with full set)
         Remove one (select which to drop)
         Add another (specify what's missing)
         Start over (different alternatives)
       ```

   - **Phase 2 — Criteria Definition**: Define comparison criteria with importance weighting.
     - Research common evaluation criteria for the domain
     - Present structured choice for criteria selection:
       ```
       "Which criteria matter most for your decision?"
       → Performance & Speed
         Ease of Use / Learning Curve
         Cost / Licensing
         Community & Ecosystem
         Scalability
       ```
     - Use sequential single-selects to build criteria list (one choice per criterion, consistent with existing structured choice protocol):
       ```
       "Include Performance & Speed as a criterion?"
       → Yes, include it
         No, skip it
       ```
     - Repeat for each candidate criterion (max 5 presented). Stop after user has selected 3-5 criteria.
     - For each selected criterion, ask importance:
       ```
       "How important is {criterion}?"
       → Critical (must-have, deal-breaker)
         Important (strongly preferred)
         Nice-to-have (would be good but not essential)
       ```

   - **Phase 3 — Structured Comparison**: Evaluate each alternative against each criterion.
     - For each criterion, present research findings across all alternatives
     - Use structured choices for user assessment:
       ```
       "For {criterion}, which alternative performs best?"
       → {Alternative A}: {evidence summary}
         {Alternative B}: {evidence summary}
         {Alternative C}: {evidence summary}
         Roughly equal (no clear winner)
       ```

   - **Phase 4 — Trade-offs & Constraints**: Surface hidden trade-offs and deal-breakers.
     - Present constraints that might eliminate options:
       ```
       "Are any of these constraints relevant?"
       → Budget limit (eliminates paid options)
         Team expertise (eliminates unfamiliar tech)
         Timeline (eliminates steep learning curves)
         None of these apply
       ```

   - **Phase 5 — Decision & Capture**: Generate comparison matrix and recommendation.
     - Score alternatives based on weighted criteria
     - Present recommendation with confidence level
     - Capture decision rationale for future reference
     - Output to `.planning/exploration-{name}.md`

2. Add compare mode deliverables:
   - Comparison Criteria (with weights: critical/important/nice-to-have)
   - Structured Comparison Matrix (alternatives × criteria)
   - Pros/Cons per Alternative
   - Trade-offs Summary
   - Risk Assessment per Alternative
   - Recommendation with Confidence Score (high/medium/low)
   - Decision Justification

3. Add compare-specific anti-patterns:
   - Don't present opinion as fact — cite research sources
   - Don't pre-select a winner — let criteria scoring drive the recommendation
   - Don't ignore user's stated constraints — re-check against each alternative
   - Don't compare more than 4 alternatives — cognitive overload

## Task 2: Add Compare Patterns to Polymath Engine

**Action**: Extend `skills/polymath-engine/SKILL.md` with compare-specific gap categories, exchange patterns, and deliverable template.

1. Add compare gap categories to Section 3 (Knowledge Gap Detection):
   - **Criteria gap**: "What factors matter most for this decision?" (importance/weight)
   - **Trade-off gap**: "Which downsides are acceptable vs. deal-breakers?"
   - **Validation gap**: "How would we verify this alternative works in our context?"
   - **Cost gap**: "What are the total costs (license, migration, maintenance)?"
   - **Experience gap**: "What do real users say about each alternative?"

2. Add compare exchange pattern to Section 4 (Exchange Management):
   - Exchange 1: Alternative identification and confirmation (2-4 options)
   - Exchange 2: Criteria selection and weighting (sequential single-selects + importance)
   - Exchange 3-4: Structured comparison per criterion (evidence-based scoring)
   - Exchange 5: Trade-offs and constraint checking
   - Exchange 6: Scoring summary and preliminary recommendation
   - Exchange 7: Decision confirmation and rationale capture
   - Max exchanges: 7 (same as other modes)

3. Add compare deliverable template to Section 5 (or new Section 5.3):
   ```markdown
   ## Comparison: {decision_topic}

   ### Alternatives Evaluated
   {list of 2-4 alternatives with brief descriptions}

   ### Comparison Criteria
   | Criterion | Weight | Description |
   |-----------|--------|-------------|
   | {criterion} | {critical/important/nice-to-have} | {what it means} |

   ### Comparison Matrix
   | Criterion | {Alt A} | {Alt B} | {Alt C} |
   |-----------|---------|---------|---------|
   | {criterion} | {score + evidence} | {score + evidence} | {score + evidence} |

   ### Pros & Cons
   #### {Alternative A}
   - Pros: {list}
   - Cons: {list}

   ### Trade-offs
   {key trade-offs that affect the decision}

   ### Risk Assessment
   | Alternative | Risk Level | Key Risks |
   |-------------|-----------|-----------|

   ### Recommendation
   **Winner**: {alternative} (Confidence: {high/medium/low})
   **Rationale**: {why this alternative scores highest on weighted criteria}

   ### Decision Captured
   - **Decided**: {date}
   - **Deciding factors**: {top 2-3 criteria that drove the decision}
   - **Rejected alternatives**: {and why}
   ```

4. Update Section 7 (State Management) with compare-specific state fields:
   - `alternatives`: list of alternatives being compared
   - `criteria`: list of {name, weight} objects
   - `scores`: matrix of alternative × criterion scores
   - `constraints`: list of active constraints
   - `recommendation`: {alternative, confidence, rationale}

**Verify**: `grep -c 'comparison.*matrix\|Comparison Matrix' skills/polymath-engine/SKILL.md` returns ≥2
