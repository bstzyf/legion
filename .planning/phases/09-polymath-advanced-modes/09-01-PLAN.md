**Action**: Modify `commands/explore.md` to present a mode selection choice before spawning the Polymath agent.

   - Add a mode selection step using `adapter.ask_user`:
     ```
     "Which exploration mode fits your goal?"
     → Crystallize (default): Transform a raw idea into a clear project concept
       Onboard: Get familiar with an existing codebase through guided exploration
       Compare: Evaluate multiple approaches side-by-side with decision capture
       Debate: Explore a question from opposing viewpoints with winner tracking

   - Store selected mode in exploration state
   - For crystallize mode: continue with existing flow unchanged

     - Onboard: "Which codebase or area do you want to explore?" (with structured choices if .planning/CODEBASE.md exists)

   - Add mode-specific briefing to agent instructions

   - Crystallize: Proceed to /legion:start | Explore more | Park (existing)
   - Onboard: Familiarized | Explore deeper | Switch to different area
   - Compare: Decision made | Need more options | Refine criteria

## Task 2: Add Onboard Mode Workflow to Polymath Agent

**Action**: Extend `agents/polymath.md` with the onboard mode workflow for guided codebase familiarization.

1. Add a "Mode Selection" section after the existing core mission, before the workflow phases:
   - Define how mode selection is received (from explore command context)
   - Define mode-specific mission statements:
     - Crystallize: "Transform raw ideas into clear project concepts" (existing)
     - Onboard: "Guide structured codebase familiarization with progressive depth"
     - Compare: "Produce structured comparison matrices with decision capture"
     - Debate: "Facilitate adversarial exploration with evidence-based winner tracking"

2. Add onboard mode workflow phases (parallel to existing crystallize phases):
   - **Phase 1 — Structure Scan**: Research the codebase structure using available context (.planning/CODEBASE.md, file tree, package files). Identify major components, entry points, and architecture patterns.
   - **Phase 2 — Depth Selection**: Present structured choice for exploration depth:
     ```
     "How deep do you want to go?"
     → Overview: High-level architecture, key directories, tech stack (3-4 exchanges)
       Architecture: Component relationships, data flow, design patterns (5-6 exchanges)
       Code Walkthrough: Specific files, implementation details, conventions (6-7 exchanges)
     ```
   - **Phase 3 — Progressive Familiarization**: Guide through the codebase at selected depth using structured choices:
     - Each exchange covers one area (e.g., "Which area interests you most?")
     - Present findings from research, ask user to confirm understanding
     - Track which areas have been covered vs. remaining
   - **Phase 4 — Knowledge Validation**: Present summary of what was covered, identify remaining gaps
   - **Phase 5 — Deliverable Generation**: Produce onboard-specific output document

3. Add onboard mode deliverables to the Deliverables section:
   - Codebase Overview (structure, languages, frameworks)
   - Key Files & Entry Points
   - Architecture Patterns Identified
   - Conventions Detected
   - Dependencies & External Services
   - Knowledge Gaps & Unknowns
   - Recommended Next Steps for Deeper Learning

4. Add onboard-specific anti-patterns:
   - Don't dump entire file contents — summarize and highlight key patterns
   - Don't assume user knows the tech stack — explain framework-specific patterns
   - Don't skip areas because they seem "obvious" — let user choose what to skip

**Verify**: `grep -c 'onboard' agents/polymath.md` returns ≥10
**Done when**: Polymath agent has a complete onboard mode workflow with progressive depth

## Task 3: Add Onboard Patterns to Polymath Engine

**Action**: Extend `skills/polymath-engine/SKILL.md` with onboard-specific gap categories, exchange patterns, and deliverable template.

   - **Convention gap**: "What patterns and naming conventions does this codebase follow?"
   - **Dependency gap**: "What external services, libraries, or APIs are required?"
   - **Setup gap**: "How do you build, run, and test this project?"
   - **History gap**: "Why were certain architectural decisions made?"

2. Add onboard exchange pattern to Section 4 (Exchange Management):
   - Exchange 1: Scope selection — full codebase vs. specific area vs. specific concern
   - Exchange 2: Depth selection — overview / architecture / code walkthrough
   - Exchange 3-4: Progressive exploration of selected areas (structured choices per area)
   - Exchange 5-6: Deep dive into user-selected topics (based on prior choices)
   - Exchange 7: Knowledge validation and gap identification
   - Max exchanges: 7 (same as crystallize)

   ```

     - `areas_remaining`: list of unexplored areas
