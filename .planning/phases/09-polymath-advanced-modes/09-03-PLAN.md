## Task 1: Add Debate Mode Workflow to Polymath Agent

**Action**: Extend `agents/polymath.md` with the debate mode workflow for adversarial exploration with winner tracking.

1. Add debate mode workflow phases (alongside crystallize, onboard, and compare phases):
   - **Phase 1 — Position Setup**: Define the two opposing positions for the debate.
     - Parse the debate topic from user input (e.g., "Should we use microservices or monolith?")
     - Research both sides to build initial position summaries
     - Present positions for user confirmation:
       ```
       "Here are the two positions I'll argue. Do these capture the debate?"
       → Yes, these are right
         Adjust Position A (refine the framing)
         Adjust Position B (refine the framing)
         Reframe entirely (different angle)
       ```

   - **Phase 2 — Evidence Gathering (Position A)**: Build the case for Position A.
     - Research evidence supporting Position A (technical, practical, precedent)
     - Present the 3-5 strongest arguments with evidence
     - Ask user to assess strength:
       ```
       "How strong is Position A's case so far?"
       → Compelling (strong evidence, hard to argue against)
         Reasonable (good points but not overwhelming)
         Weak (evidence is thin or unconvincing)
         Mixed (some strong points, some weak)
       ```
     - Map assessment to preference signal:
       - Compelling or Reasonable → Position A preferred (+1 to A)
       - Weak → Position B preferred (+1 to B, since A's case was weak)
       - Mixed → Tie (+0.5 to each)

   - **Phase 3 — Evidence Gathering (Position B)**: Build the case for Position B.
     - Research evidence supporting Position B (same rigor as Position A)
     - Present the 3-5 strongest arguments with evidence
     - Ask user to assess strength (same scale as Phase 2)
     - Map assessment to preference signal (same mapping, reversed: Compelling/Reasonable → B preferred, Weak → A preferred, Mixed → Tie)

   - **Phase 4 — Counter-Arguments**: Each position responds to the other's evidence.
     - Present Position A's counter-arguments to Position B's evidence
     - Present Position B's counter-arguments to Position A's evidence
     - Ask user which counter-arguments land:
       ```
       "Which position made better counter-arguments?"
       → Position A countered more effectively
         Position B countered more effectively
         Both made equally strong counters
         Neither addressed the other's points well
       ```
     - Map to preference signal:
       - Position A countered more effectively → A preferred (+1 to A)
       - Position B countered more effectively → B preferred (+1 to B)
       - Both equally strong or neither addressed → Tie (+0.5 each)

   - **Phase 5 — Scoring & Winner Declaration**: Compute winner using DPO-inspired scoring.
     - Tally preference signals across exchanges:
       - Each exchange where user preferred Position A: +1 to A
       - Each exchange where user preferred Position B: +1 to B
       - Ties: +0.5 to each
     - Compute win probability: P(A wins) = score_A / (score_A + score_B)
     - Present results:
       ```
       "Debate Results:"
       Position A: {score} / {total} ({percentage}%)
       Position B: {score} / {total} ({percentage}%)
       Winner: {position} (Confidence: {high if >70%, medium if 50-70%, low if <50%})
       ```
     - If tie or low confidence, offer tiebreaker exchange
     - Capture winner, reasoning, and key evidence in deliverable

2. Add debate mode deliverables:
   - Position A Summary (framing, key arguments)
   - Position B Summary (framing, key arguments)
   - Evidence for Position A (3-5 arguments with sources)
   - Evidence for Position B (3-5 arguments with sources)
   - Counter-Arguments (each side's response to the other)
   - Scoring Breakdown (per-exchange preference signals)
   - Winner & Confidence Level
   - Remaining Uncertainties
   - Recommended Next Actions

3. Add debate-specific anti-patterns:
   - Don't argue one side more passionately than the other — maintain adversarial balance
   - Don't present strawman arguments for the "losing" side — both positions deserve best case
   - Don't let the user's initial lean bias the evidence gathering — research both sides equally
   - Don't skip counter-arguments — they're the most valuable part of the debate
   - Don't declare a winner without user preference signals — DPO requires human feedback

## Task 2: Add Debate Patterns to Polymath Engine

**Action**: Extend `skills/polymath-engine/SKILL.md` with debate-specific gap categories, exchange patterns, and deliverable template.

1. Add debate gap categories to Section 3 (Knowledge Gap Detection):
   - **Position gap**: "What's the strongest argument for the opposing side?"
   - **Evidence gap**: "What facts or data support each position?"
   - **Assumption gap**: "What are we assuming that might be wrong?"
   - **Precedent gap**: "Who has tried each approach, and what happened?"
   - **Blind spot gap**: "What are we not considering that could change the outcome?"

2. Add debate exchange pattern to Section 4 (Exchange Management):
   - Exchange 1: Position setup and confirmation (define A vs B)
   - Exchange 2: Evidence for Position A (research + present + assess)
   - Exchange 3: Evidence for Position B (research + present + assess)
   - Exchange 4: Counter-arguments (both directions + preference signal)
   - Exchange 5: Blind spots and assumptions (surface hidden factors)
   - Exchange 6: Scoring summary and winner declaration
   - Exchange 7: Tiebreaker (only if needed) or next steps
   - Max exchanges: 7 (same as other modes)

3. Add DPO-inspired scoring mechanism documentation:
   - **Preference signals**: Binary per exchange (A preferred, B preferred, or tie)
   - **Assessment-to-signal mapping**:
     - For evidence exchanges: Compelling/Reasonable → that position preferred; Weak → opposing preferred; Mixed → Tie
     - For counter-argument exchange: "countered more effectively" → that position preferred; equal/neither → Tie
     - For blind spots exchange: "reveals more risk for X" → opposing position preferred; equal → Tie
   - **Scoring**: Simple tally — each preference = 1 point to preferred position, tie = 0.5 each
   - **Win probability**: P(A) = score_A / (score_A + score_B)
   - **Confidence thresholds**:
     - High: >70% win probability (clear winner)
     - Medium: 50-70% win probability (leaning winner)
     - Low: <50% or exactly 50% (no clear winner, suggest tiebreaker)
   - **Why DPO-inspired**: Uses human preference signals (not LLM judgment) to determine winner, avoiding model bias. The user decides which arguments land, not the AI.

4. Add debate deliverable template to Section 5 (or new Section 5.4):
   ```markdown
   ## Debate: {topic}

   ### Position A: {position_a_name}
   **Framing**: {how this position sees the question}
   **Key Arguments**:
   1. {argument + evidence}
   2. {argument + evidence}
   3. {argument + evidence}

   ### Position B: {position_b_name}
   **Framing**: {how this position sees the question}
   **Key Arguments**:
   1. {argument + evidence}
   2. {argument + evidence}
   3. {argument + evidence}

   ### Counter-Arguments
   #### Position A's response to B
   {counter-arguments}

   #### Position B's response to A
   {counter-arguments}

   ### Scoring
   | Exchange | Preference | Signal |
   |----------|-----------|--------|
   | Evidence A | {assessment} | A+1 / B+1 / Tie |
   | Evidence B | {assessment} | A+1 / B+1 / Tie |
   | Counters | {assessment} | A+1 / B+1 / Tie |
   | Blind spots | {assessment} | A+1 / B+1 / Tie |

   **Score**: Position A: {x}/{total} | Position B: {y}/{total}
   **Winner**: {position} (Confidence: {high/medium/low})

   ### Remaining Uncertainties
   {what could change the outcome}

   ### Next Actions
   {recommended follow-up based on debate outcome}
   ```

5. Update Section 7 (State Management) with debate-specific state fields:
   - `position_a`: {name, framing, arguments[]}
   - `position_b`: {name, framing, arguments[]}
   - `evidence`: {position_a: [], position_b: []}
   - `counter_arguments`: {a_to_b: [], b_to_a: []}
   - `preference_signals`: [{exchange, preferred, signal}]
   - `scores`: {position_a: number, position_b: number}
   - `winner`: {position, confidence, rationale}
